Part 1 — オニオンアーキテクチャとは何か

公開日: 2008年7月29日

「オニオンアーキテクチャ」と名付けた設計手法について、これまで何度も講演してきました。 システム全体を通じて関心の分離を徹底することで、より保守しやすいアプリケーションが生まれると確信しています。

この設計が向いている場面： 小規模なウェブサイトには向いていません。長期運用が前提のビジネスアプリや複雑な業務ロジックを持つシステムに適しています。インターフェースを使った契約ベースの設計と、インフラの外部化を強制します。

従来型レイヤードアーキテクチャの問題

最も多く見られる従来型の設計は、各層が下の層に依存する「積み重ね」構造です。

図1: 従来型レイヤードアーキテクチャ — UIはデータアクセスに間接的に依存してしまう

この構造の最大の問題は、UIとビジネスロジックが、データアクセスに結合してしまう点です。 直接ではなくても、間接的な依存（推移的依存）は依然として依存です。

歴史的に見て、データアクセス技術は3年ごとに変化してきました。 長期運用するシステムでは、3年後にはデータアクセスの改修が必要になると考えてよいでしょう。 結合がきつければ改修が困難になり、システムが老朽化したまま放置されます。 これがレガシーシステムが生まれる根本原因の一つです。

オニオンアーキテクチャの基本構造

新しいアプローチを提案します。完全に新しいわけではありませんが、 名前のついたアーキテクチャパターンとして定義します。 名前があることで、エンジニア同士が共通の語彙で議論できるようになります。

図2: オニオンアーキテクチャ — 依存はすべて外側から中心（内側）へ向かう

最も中心にあるのがドメインモデル（その組織・業務の「状態」と「振る舞い」を表したオブジェクト群）です。 すべての依存が中心に向かうため、ドメインモデルは自分自身にしか依存しません。

ドメインモデルの外側には、オブジェクトの保存・取得の振る舞いを定義するリポジトリインターフェースがあります。 「データをどうやって保存・取得するか」の約束だけを定義し、実際にどのDBを使うかは外側の層が担当します。 中心部はDBの存在を知りません。

「DBは中心ではない、外部にあるものだ」 — これはオニオンアーキテクチャの核心です。 従来の「DBアプリ」という発想を捨てる必要があります。 DBやファイルシステムからアプリを切り離すことで、長期的な保守コストが大幅に下がります。

このアーキテクチャは依存性逆転の原則（DIP）に強く依拠しています。 「上位モジュールは下位モジュールに依存してはいけない。どちらも抽象（インターフェース）に依存すべき」という設計原則です。 アプリコアはインターフェースの実装を必要としますが、その実装クラスは外側の端に置かれます。 実行時に必要なクラスを注入するためにIoCコンテナ（DIコンテナとも呼ばれる）を使います。

なお、Alistair Cockburn のヘキサゴナルアーキテクチャ（ポート&アダプター）も「インフラを外部化し、アダプターコードを書くことで密結合を防ぐ」という同じ前提を持ちます。

Part 2 — 実践例：CodeCampServer

公開日: 2008年7月30日

実際のプロジェクト「CodeCampServer」を例に、オニオンアーキテクチャがどのように動くかを示します。 ASP.NET MVC を使用した実装例ですが、設計概念は言語非依存です。

登場クラスとその位置づけ

クラス / インターフェース どの層？ 役割
SpeakerController 外側（UI層） 画面処理。フレームワークに依存
IConferenceRepository 内側（コア） データ取得の「約束」だけ定義
IUserSession 内側（コア） ユーザーセッションの「約束」
ConferenceRepository 外側（インフラ層） IConferenceRepository の実装。DB処理を持つ
UserSession 外側（インフラ層） IUserSession の実装

SpeakerController はインターフェース（約束）だけに依存しています。どんなDBを使うかは一切知りません。

// SpeakerControllerのコンストラクタ（初期化メソッド）
public SpeakerController(
    IConferenceRepository conferenceRepository,  // インターフェース（約束）を受け取る
    IUserSession userSession,
    IClock clock)
{
    _conferenceRepository = conferenceRepository;
    _clock = clock;
    _userSession = userSession;
}

実行時の仕組み（IoCコンテナ）： アプリ起動時にIoCコンテナが「IConferenceRepository が必要なら ConferenceRepository を渡せ」という設定を読み込み、 自動的に実装クラスを注入します。SpeakerController は実装の詳細を一切知らなくて済みます。

ルールの確認として、SpeakerController は同じ層にある UserSession を直接使えます（同層はOK）。 しかしConferenceRepository は使えません（外側のクラスを内側から参照することになるため）。 必ず外から IConferenceRepository として渡してもらう必要があります。

Part 3 — 従来型との比較と4つの原則

公開日: 2008年8月4日

Part 3 の目的は、オニオンアーキテクチャと従来型レイヤードアーキテクチャを対比させることです。 なお、私は技術的なブレークスルーを主張しているわけではありません。 Martin Fowler、Kent Beck らに学んだことを統合し、コミュニケーションのための名前を付けたものです。

インフラの扱い方が根本的に違う

インフラとは「汎用的で、アプリに競争優位をもたらさないコード」（データアクセス、ファイルI/O、外部WebAPIなど）のことです。 このコードは変化しやすいのに、従来型ではビジネスロジックがインフラに直接結合してしまいます。

観点 従来型レイヤード オニオン
インフラの位置 ビジネスロジックの下に積み重なる 外側の端に追いやられる
ビジネスロジックとDBの結合 直接依存（結合強） インターフェース越しのみ（結合弱）
層をまたいだ呼び出し 直下の層にしか呼べない どの内側の層でも直接呼べる
テストコードの位置 定義が曖昧 外側の端（コアに依存しない）
DBなしでのコンパイル 困難 可能（コアはDBを知らない）

「外側の層はどの内側の層でも直接呼べる」 — これがオニオンの重要な特徴です。 従来型は「UIはビジネスロジックのみ呼べる。ビジネスロジックはDAのみ呼べる」という制約がありました。 オニオンはそれを廃止し、依存の方向さえ守れば内側なら何でも呼べます。 これにより不要な中継メソッドが減ります。

オニオンアーキテクチャの4つの原則

Part 3 の最後に、Palermo は4原則を明文化しました。これがオニオンアーキテクチャの定義です。

  • 原則1：アプリは独立したオブジェクトモデルを中心に構築する
    ドメインモデルはDBにもフレームワークにも依存しない。業務の概念をそのままコードで表現したものが中心にいる。
  • 原則2：内側の層がインターフェースを定義し、外側の層が実装する
    「何をするか」の約束はコアが持ち、「どうやってするか」の実装は外側が担う。内側は外側の実装を知らない。
  • 原則3：依存の方向は必ず中心に向かう
    外側から内側への依存はOK。内側から外側への依存は絶対NG。これが崩れるとオニオンではなくなる。
  • 原則4：アプリコアはインフラなしでコンパイル・実行できる
    DBが繋がらなくてもビジネスロジックのテストができる。コアが本当に独立していることの証明になる。

IoCコンテナは必須ではありません。 4原則にIoCコンテナは含まれていません。C# や Java なら便利ですが、言語によってはDI機能が組み込まれています。 オニオンアーキテクチャはIoCなしでも成立します。

この構造により、以下が変わってもアプリコアは変更不要になります。

  • UIの変更
  • データアクセス技術の変更
  • Webサービス・メッセージング基盤の変更
  • I/O技術の変更

Part 4 — 4年後の振り返り

公開日: 2013年8月19日

2008年に「オニオンアーキテクチャ」という名前を作ってから4年が経ちました。 ユーザーグループや各種カンファレンスで講演し、Manning社の書籍「ASP.NET MVC in Action」の一章にも掲載されました。 このパターンが広く受け入れられたことに驚いています。

よく聞かれる質問への回答

「DDDと組み合わせないといけないの？」

DDD（ドメイン駆動設計）（Eric Evansが提唱した、業務の概念をそのままコードに落とし込む設計手法）がなくてもオニオンアーキテクチャは機能します。 CQRS（書き込み処理と読み込み処理を別々のモデルで扱う手法）とも、フォーム中心の設計とも、DDDとも相性が良い。 オニオンはあくまで「コアオブジェクトモデルが不安定なコードに依存しない」という構造のパターンです。

4年経っても変わらない4つの原則

  • アプリは独立したオブジェクトモデルを中心に構築する
  • 内側の層がインターフェースを定義し、外側の層が実装する
  • 依存の方向は必ず中心に向かう
  • アプリコアのコードはインフラなしでコンパイル・実行できる

多くの方から「さまざまな環境での実装方法」を聞かれました。 そこで、よりシンプルなサンプルコードを公開しています。UIの入力フォームやIoCコンテナをあえて外しており、 オニオンアーキテクチャの本質だけを伝えることを目指しています。

GitHubに有志がフォークしたサンプルが残っています： github.com/Jordiag/Jeffrey-Palermo-Onion-Architecture

また、同様の思想を持つアーキテクチャとして、Alistair Cockburn による Hexagonal Architecture（ポート&アダプター）があります。 オニオンはそれをベースにしながら、内部の層を明示的に整理したものです。 後の Robert C. Martin のクリーンアーキテクチャもこれらの延長線上にあります。

依存関係の流れ（正しい方向）

図3: 依存の向きは必ず中心（右）に向かう。逆方向はない。

覚えておくべき3つのポイント

ポイント 意味
DBは外側にある DBはアプリの中心ではなく、交換可能な外部ストレージ。インターフェース越しにのみ接触する。
約束は内側、実装は外側 インターフェース（何をするか）はコアが持つ。具体的な処理（どうやるか）は外側が書く。
依存は中心方向のみ 外→内はOK。内→外は絶対NG。これだけ守ればオニオンの骨格は成立する。
