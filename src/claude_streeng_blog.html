Claude Code を操る：スキル、フック、ルール、サブエージェントほか

Claude の動作を指示する7つの方法を学び、それぞれのコンテキストコストと権限を理解しましょう。

Claude はあなたの作業スタイルに合わせて動作するよう設計されており、Claude Code ではその動作をカスタマイズできます。

Claude の動作を指示する方法は7つあります：CLAUDE.md ファイル、ルール、スキル、サブエージェント、フック、出力スタイル、そしてシステムプロンプトへの追記です。

各方法は以下の3点を制御します：

  • 指示がコンテキストにロードされるタイミング
  • 長いセッションを通じて保持されるかどうか（コンパクション動作）
  • どれだけの権限を持つか

下表は各方法の主な違いをまとめたものです。各方法の詳細と、どの指示をどこに置くべきかの判断フレームワークについては後述します。

指示を届ける7つの方法

CLAUDE.md ファイル

CLAUDE.md はプロジェクトのルートに置くマークダウンファイルです。セッション開始時にコンテキストにロードされ、セッション全体を通じて保持されます。

ビルドコマンド、ディレクトリ構成、モノレポ構造、コーディング規約、チームの規範はここに自然にフィットします。

ロードのされ方によって2種類あります：

  • 常時ロード：ルート CLAUDE.md ファイル（共有リポジトリ内、またはプロジェクト固有の個人設定としてローカルに保存）。これらはセッション開始時にすべてロードされ、長いセッションでも失われたり劣化したりしません。Claude Code が会話をコンパクトにするとき、これらのファイルを再読み込みします。
  • オンデマンド：セッションを開始したフォルダより下のサブディレクトリにある CLAUDE.md ファイル。例えば app/api/CLAUDE.md は、Claude が app/api 下のファイルを読んだときにロードされ、セッション開始時ではありません。パススコープのルールと同じコンパクション動作を取り、そのサブディレクトリが再度触れられるまで消失します。

共有リポジトリでは CLAUDE.md が所有者不在の設定ファイルのように肥大化しがちです。各チームが自分の指示を追加し、何も削除されません。そのコストはスケールすると増大します。

リポジトリで作業するすべてのエンジニアのすべてのセッションに、タスクとの関連性に関わらずすべての行がロードされ、トークンを消費し、本当に重要な指示への遵守を希薄化させます。ファイルが肥大化してきたら、チーム固有の規約はパススコープのルールに、手順はスキルに移して、関連するときのみロードされるようにしましょう。

💡 ヒント

CLAUDE.md は200行以内に収め、オーナーを設けて、コードのようにレビューしましょう。このファイルは Claude にコードベースの概要を伝えるもの、または必要に応じてより詳細な情報を見つけられる他のファイルへのインデックスとして考えましょう。

モノレポでは各チームのディレクトリに独自のサブディレクトリ CLAUDE.md を持たせることで、チームは自分たちの規約のみをロードできます。また開発者は claudeMdExcludes 設定を使って、決して触れないチームのファイルをスキップできます。

セキュリティポリシーやコンプライアンス要件など、組織内のすべてのリポジトリに適用される基準は、MDM や設定管理ツールを通じて開発者のマシンにデプロイできる中央管理の CLAUDE.md に配置でき、個人の設定では除外できません。

ルール

ルールは .claude/rules/ 内のマークダウンファイルで、Claude に特定の制約や規約を与えます。

スコープなしのルールは CLAUDE.md と同様に、セッション開始時に常時ロードされ、コンパクション時に再注入されます。これにより、手元のタスクに関連しない場合でもコンテキストがロードされ、トークンを無駄にする可能性があります。

パススコープのルールを使うと、paths フィールドを追加して関連するときのみ命令がロードされるようにできます。

例えば、src/api/** にスコープされたルールは、ドキュメントのみのセッション中はコンテキスト外にとどまります。src/api/ ディレクトリ内のファイルを Claude が読んだときにのみロードされます。 

---
paths:
  - "src/api/**"
  - "**/*.handler.ts"
---
すべての API ハンドラーは処理前に Zod で入力を検証しなければなりません。
💡 ヒント

「マイグレーションは追記のみ」のようなファイル固有の制約は、paths: のフロントマターに置いたルールとして最も適しています。コードベースの複数の（ただしすべてではない）場所に現れる横断的な関心事や、ネストされた CLAUDE.md ファイルでは適切でないファイルに指示が関係する場合は、パススコープのルールを選びましょう。

スキル

スキルは .claude/skills/ に、命令・スクリプト・リソースのフォルダとして置かれ、Claude が動的にロードします。各スキルには名前・説明・本文を持つ SKILL.md ファイルがあります。

セッション開始時には名前と説明のみがロードされ、スラッシュコマンド（/code-review）またはタスクの自動マッチングでスキルが呼び出されたときに、本文全体がロードされます。

例えば、/code-review は組み込みスキルで、現在の差分をレビューし、ファイルを編集せずに結果を報告します。スキルがプレイブックを定義するため、Claude は呼び出すたびに同じ構造化されたアプローチに従います。

コンパクション時、Claude Code は呼び出されたスキルをすべての呼び出し済みスキル間の合計バジェット内で再注入します。セッション中に多くのスキルを呼び出した場合、最も古いものから順に削除されます。

💡 ヒント

デプロイワークフロー、リリースチェックリスト、レビュープロセスのような手順的な指示は、CLAUDE.md ではなくスキルに置きましょう。

サブエージェント

サブエージェントは .claude/agents/ 内のマークダウンファイルで、特定のサイドタスク用の隔離されたアシスタントを定義します。各ファイルは YAML フロントマター（名前・説明・オプションのモデルとツールアクセスのフィールド）と、そのサブエージェントのシステムプロンプトになる本文で構成されます。

サブエージェントはスキルと似ており、セッション開始時に名前・説明・ツールリストがロードされますが、エージェントの本文内の大きなコンテキストは自動呼び出しされません。Claude は Agent ツールを通じてプロンプト文字列を渡して呼び出します。

サブエージェントの本文内の大きな命令コンテキストが自動呼び出しされないだけでなく、親会話には一切入りません。

サブエージェントはその独自の新鮮なコンテキストウィンドウで実行され、メインセッションに返却されるのは、サブエージェントの最終メッセージ（多くの場合、多くのサブタスクの集約結果）とメタデータのみです。

このパターンはスケールします：サブエージェントは最大5レベルまでネストでき、ダイナミックワークフローは、サブエージェントアーキテクチャの各詳細を指定することなく、数十から数百のバックグラウンドエージェントをオーケストレーションします。オーケストレーション計画と中間結果は Claude のコンテキストウィンドウではなくスクリプト変数に保存されるため、命令の忠実度を失わずにスケールできます。

💡 ヒント

その隔離性がスキルではなくサブエージェントを選ぶ主な理由の一つです。詳細検索、ログ解析パス、依存関係監査のようなサイドタスクが、二度と参照しない中間結果でメイン会話を散らかすような場合は、サブエージェントを使いましょう。各ステップを見て操縦できるようにメインスレッドで手順を実行したい場合はスキルを使いましょう。

フック

フックはユーザー定義のコマンド・HTTP エンドポイント・LLM プロンプトで、ファイル編集・ツール呼び出し・セッション開始など、Claude のライフサイクルの特定のイベントで発火することで、Claude の動作をより確定的に制御します。

フックは settings.json・管理ポリシー設定・スキル/エージェントのフロントマターに登録します。

フックにはいくつかの種類があります：command、HTTP、mcp_tool、prompt、agent です。最初の3つは確定的に実行され、prompt と agent はルールの集合ではなく Claude の判断を使って出力を決定します。

フックの設定や命令はメインコンテキストウィンドウの外に存在するため、コンテキストコストは低くなります。ハーネスはフックの種類に応じてハンドラー（command、http、mcp_tool）を実行するか、別のウィンドウでモデル呼び出しを行います（prompt、agent）。

一部のフックは出力がメインコンテキストウィンドウに保存される場合があります。例えば、ブロッキングフックの標準エラー出力は、なぜその呼び出しが拒否されたかを Claude が知るためにコンテキスト内に保存されます。

しかし、設定が明示的に返却しない限り、ほとんどのフックはメインウィンドウに出力を保存しません。PreCompact イベントを使ってコンパクション前にチャット履歴を別ファイルにバックアップした場合、Claude はどのファイルにチャット履歴が保存されたかを知りません。

💡 ヒント

確定的に実行されるべきことすべてにフックを使いましょう：編集後のリンター実行、完了時の Slack 投稿、コマンドの事前ブロックなど。PreToolUse フックは任意のツール呼び出しを検査し、終了コード 2 で拒否できます。コンテキストにロードされる命令ではなく、ハーネスが実行するコードであるため、コンテキストコストが低くなります。

出力スタイル

出力スタイルは .claude/output-styles/ 内のファイルで、システムプロンプトに命令を注入します。コンパクションされず、すべてのセッション開始時にロードされ、セッション内の最初のリクエスト後にキャッシュされるため、コンテキストコストは中程度です。

システムプロンプトに置かれるため、出力スタイルはこれまで紹介した方法の中で最も高い命令遵守の重みを持ち、慎重に使う必要があります。

出力スタイルへの変更はデフォルトの出力スタイルを置き換えます（スタイルのフロントマターで keep-coding-instructions: true を設定しない限り）。

Claude Code では、これによって Claude がソフトウェアエンジニアリングタスクを支援していることを伝え、以下のような重要なデフォルト命令を含む指示が削除されます：

  • 変更のスコープの限定方法
  • コードコメントを追加または省略するタイミング
  • セキュリティの懸念への対応方法
  • 作業完了を宣言する前にテストを実行するなどの確認習慣

デフォルトでは、カスタム出力スタイルはこれらすべてを削除し、Claude Code はソフトウェアエンジニアアシスタントではなく、汎用アシスタントに近くなります。

💡 ヒント

カスタム出力スタイルを書く前に、組み込みスタイルを確認しましょう。Proactive（自律的）、Explanatory（説明的）、Learning（学習向け）は最も一般的なニーズ（自律性・教育モード・協調コーディング）をカバーしており、スタイルファイルを管理する必要がありません。

システムプロンプトへの追記

出力スタイルファイルを変更する代わりに、append-system-prompt フラグを使う方法があります。出力スタイルファイルを変更すると Claude の動作に大きく意図しない変化が起きる可能性があるのに対し、追記フラグは元のシステムプロンプトに対して加算的なものです。Claude の役割を変更せず、デフォルトの役割に命令を追加するだけです。

また呼び出し時に渡されるため、ファイルとしてセッションをまたいで保持されるのではなく、その呼び出しにのみ適用されます。

システムプロンプトへの追記は他の方法と比べてコンテキストコストが高くなる場合があります。入力トークンが増加しますが、セッション内の最初のリクエスト後はプロンプトキャッシュでコストが削減されます。より詳細または長いスタイルで出力するよう指示すると、出力トークンも増加します。

💡 ヒント

システムプロンプトへの追記は、特定のコーディング標準、出力フォーマット、ドメイン固有の知識の追加に最適です。この方法は提供する命令が増えるほど遵守の効果が薄れることを念頭に置いてください。特に矛盾する命令がある場合、Claude はより厳密でなくなります。

Claude Code カスタマイズのクイックヒント

以下のことをしていると気づいたら、指示の置き場所を変更することを検討してください：

CLAUDE.md に「X のたびに、必ず Y せよ」と書く場合。編集後に prettier を実行したり完了時に Slack に投稿したりするような、確実に実行されるべき動作は、代わりに settings.json のフックを使いましょう。Claude がフォーマッターを実行することを選択するのと、フォーマッターが自動的に実行されるのは別物です。

CLAUDE.md に「絶対にこれをするな」と書く場合。絶対に起きてはならないことがある場合、命令は適切なツールではありません。Claude はほとんどの場合に命令に従いますが、プレッシャー下、長いセッション、曖昧な状況、またはタスクの一部でアクセスされたファイル内のプロンプトインジェクションによって、Claude はプロンプトされたルールに従えない場合があります。本物のガードレールは確定的である必要があり、その強制方法はフックとパーミッションです。PreToolUse フックは呼び出しを検査してコード 2 で終了することで拒否できます。管理設定はさらに踏み込みます：管理者がデプロイし、ユーザーのローカル設定では上書きできず、確定的な組織全体のガードレールを強制する唯一の方法です。

CLAUDE.md に30行の手順を書く場合。手順はスキルに属します。CLAUDE.md は Claude が常に保持すべき事実のためのものです：ビルドコマンド、モノレポのレイアウト、チームの規約。デプロイのランブックやセキュリティレビューチェックリストは .claude/skills/ に置き、呼び出されたときのみ本文がロードされるようにするべきです。

パスなしの API 固有のルールを書く場合。ルールが src/api/** にのみ適用されるなら、paths: でスコープすることで、無関係の作業中はコンテキスト外に保てます。スコープなしのルールは、コンテキストに常時ロードされてトークンを消費するという点で、CLAUDE.md にコンテキストを書くのと機械的に同じです。

プロジェクトレベルの CLAUDE.md に個人の好みを書く場合。すべてのファイルベースの方法には、どのリポジトリにいても、すべての Claude Code セッションでロードされるユーザーレベルの対応物があります。個人の好みにはローカルファイルを使いましょう（常にセマンティックコミットメッセージを使うなど）。プロジェクトレベルのファイルは、特定のコードベースに固有のチーム全体の好みに使いましょう。

はじめに

環境の設定から並行セッションへのスケールまで、Claude Code を最大限に活用するためのヒントとパターンは、Claude Code のベストプラクティスドキュメントで確認できます。

これらのいくつかが動作するようになったら、スキル・サブエージェント・フック・出力スタイルをプラグインとしてバンドルして、チームメンバーやプロジェクト全体で一貫したセットアップを共有できます。
