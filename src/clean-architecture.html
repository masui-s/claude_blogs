クリーンアーキテクチャ入門

2026-06-18

クリーンアーキテクチャとは

クリーンアーキテクチャは、Robert C. Martin（Uncle Bob）が提唱したソフトウェア設計の原則です。ビジネスロジックをフレームワーク・データベース・UI などの外部要素から独立させることを目的としています。

核心的な考え方は 「依存関係ルール（Dependency Rule）」 です。ソースコードの依存方向は常に内側（ビジネスルール）に向かわなければなりません。

4つの同心円

クリーンアーキテクチャは通常、4つの層として表現されます：

1. Entities（エンティティ）

最も内側の層。ビジネスルールを表すオブジェクトです。企業全体で共有される最重要のビジネスロジックを含み、外部の変化（DBの変更・UIの変更）に影響されません。

2. Use Cases（ユースケース）

アプリケーション固有のビジネスルールを担います。エンティティへの入出力フローを調整し、ユーザーの目標達成を助けるロジックをここに集約します。

3. Interface Adapters（インターフェースアダプター）

ユースケースやエンティティが扱いやすい形式と、外部（DB・Web・UI）が扱う形式の間でデータを変換します。Controller・Presenter・Gateway などがこの層に属します。

4. Frameworks & Drivers（フレームワーク・ドライバー）

最も外側の層。Webフレームワーク・データベース・UIなど、具体的な実装の詳細が含まれます。この層にはなるべく少ないコードしか書かないことが推奨されます。

依存関係ルール

内側の円は外側の円について何も知らない

外側の層の名前・関数・変数などを内側の層が参照してはいけません。データを外から内に渡す場合も、内側が定義したインターフェース（DTO）経由で行います。

依存性逆転の原則（DIP）

クリーンアーキテクチャの実現にはDIPが欠かせません。例として、ユースケース層からDBに依存する場合を考えます：

# 悪い例：Use Case が DB 実装に直接依存
class CreateUserUseCase:
    def __init__(self):
        self.db = MySQLUserRepository()  # 具体実装に依存

# 良い例：Use Case はインターフェースに依存
class CreateUserUseCase:
    def __init__(self, user_repo: UserRepository):  # 抽象に依存
        self.user_repo = user_repo

インターフェース（UserRepository）はユースケース層で定義し、MySQLの実装はアダプター層に置きます。こうすることで依存の矢印が内側を向きます。

メリットとトレードオフ

  • テスタビリティ：ビジネスロジックをDBや外部サービスなしでテストできる
  • 技術の交換可能性：DBやフレームワークを差し替えてもビジネスロジックは無傷
  • 独立したデプロイ：層を独立してデプロイ・更新できる

一方で、小規模プロジェクトでは過剰設計になりがちです。チームの規模・プロダクトのライフサイクルを考慮して採用を判断しましょう。
