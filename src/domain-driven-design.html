ドメイン駆動設計（DDD）入門

2026-06-18

DDDとは何か

ドメイン駆動設計（Domain-Driven Design、DDD）は、Eric Evans が2003年に著書『Domain-Driven Design: Tackling Complexity in the Heart of Software』で提唱した設計手法です。

中心的な考え方は 「ソフトウェアの核心はドメイン（業務領域）の複雑さに対処することにある」 という点です。技術的な実装よりも、解決すべき問題領域をモデル化することを最優先に置きます。

ユビキタス言語

DDDで最も重要な実践のひとつが ユビキタス言語（Ubiquitous Language） です。開発者とドメインエキスパート（業務担当者）が同じ言葉でコミュニケーションし、その言葉をそのままコードに反映させます。

「翻訳」のコストをなくす。業務用語がそのままクラス名・メソッド名になる。

例えば、EC サイト開発において「注文を確定する」という業務用語があれば、コード上も order.confirm() とすべきであり、order.setStatus("CONFIRMED") のような技術的表現は避けます。

境界づけられたコンテキスト

境界づけられたコンテキスト（Bounded Context） は、特定のドメインモデルが有効な範囲を明示的に定義した境界です。

同じ「顧客」という概念でも、販売コンテキストでは「購買履歴・与信情報」が重要であり、配送コンテキストでは「住所・連絡先」が中心になります。これらを無理に統合せず、コンテキストごとに独立したモデルを持たせます。

# 販売コンテキストの顧客
class Customer:
    def __init__(self, id: CustomerId, credit_limit: Money):
        self.id = id
        self.credit_limit = credit_limit

# 配送コンテキストの顧客（別モデル）
class ShippingRecipient:
    def __init__(self, id: RecipientId, address: Address):
        self.id = id
        self.address = address

エンティティと値オブジェクト

エンティティ（Entity）

エンティティは 同一性（Identity） によって識別されるオブジェクトです。属性が変化しても、IDが同じであれば同一のオブジェクトとみなします。

class Order:
    def __init__(self, order_id: OrderId, customer_id: CustomerId):
        self.order_id = order_id      # 同一性の根拠
        self.customer_id = customer_id
        self.items: list[OrderItem] = []
        self.status = OrderStatus.DRAFT

    def __eq__(self, other):
        if not isinstance(other, Order):
            return False
        return self.order_id == other.order_id  # IDで比較

値オブジェクト（Value Object）

値オブジェクトは 属性の値 によって同一性を判断するオブジェクトです。不変（イミュータブル）であり、同じ値を持つ2つのオブジェクトは等価とみなします。

from dataclasses import dataclass

@dataclass(frozen=True)
class Money:
    amount: int
    currency: str

    def add(self, other: "Money") -> "Money":
        if self.currency != other.currency:
            raise ValueError("通貨が一致しません")
        return Money(self.amount + other.amount, self.currency)

# 使用例
price1 = Money(1000, "JPY")
price2 = Money(1000, "JPY")
print(price1 == price2)  # True — 値で比較

集約（Aggregate）

集約は、関連するエンティティと値オブジェクトをひとつの整合性境界としてまとめたものです。集約の外部からは 集約ルート（Aggregate Root） を通じてのみ操作します。

class Order:  # 集約ルート
    def add_item(self, product_id: ProductId, quantity: int, unit_price: Money):
        # 集約内のビジネスルールをここで守る
        if self.status != OrderStatus.DRAFT:
            raise DomainError("確定済みの注文には商品を追加できません")
        item = OrderItem(product_id, quantity, unit_price)
        self.items.append(item)

    def confirm(self):
        if not self.items:
            raise DomainError("商品が1つもない注文は確定できません")
        self.status = OrderStatus.CONFIRMED

外部から OrderItem を直接操作することは許可されません。Order 経由でのみ変更を行うことで、集約全体の整合性が保証されます。

ドメインサービスとリポジトリ

ドメインサービス

特定のエンティティや値オブジェクトに自然に属さないビジネスロジックは ドメインサービス として切り出します。

class PricingService:
    def calculate_total(self, order: Order, tax_rate: float) -> Money:
        subtotal = sum(
            item.unit_price.amount * item.quantity
            for item in order.items
        )
        tax = int(subtotal * tax_rate)
        return Money(subtotal + tax, "JPY")

リポジトリ（Repository）

リポジトリは集約の永続化と取得を担います。ドメイン層はリポジトリのインターフェースのみを知り、DBの実装詳細には依存しません。

from abc import ABC, abstractmethod

class OrderRepository(ABC):  # ドメイン層で定義
    @abstractmethod
    def find_by_id(self, order_id: OrderId) -> Order | None: ...

    @abstractmethod
    def save(self, order: Order) -> None: ...

# インフラ層で実装
class SqlOrderRepository(OrderRepository):
    def find_by_id(self, order_id: OrderId) -> Order | None:
        row = self._db.query("SELECT * FROM orders WHERE id = ?", order_id.value)
        return self._to_entity(row) if row else None

    def save(self, order: Order) -> None:
        self._db.upsert("orders", self._to_record(order))

戦略的設計と戦術的設計

DDDは2つのレベルに分かれます：

  • 戦略的設計：境界づけられたコンテキストの識別、コンテキストマップによるシステム全体の構造設計
  • 戦術的設計：エンティティ・値オブジェクト・集約・ドメインサービス・リポジトリといった実装パターンの適用

小規模プロジェクトではまず戦略的設計からコンテキストを正しく切ることが、後々の複雑さを最も効果的に抑えます。

DDDを導入すべきか

DDDはすべてのプロジェクトに適しているわけではありません。効果が高い条件は以下のとおりです：

  • 業務ルールが複雑で頻繁に変化する
  • ドメインエキスパートと継続的に協働できる体制がある
  • 長期間にわたって保守するプロダクトである

CRUD中心の単純なシステムや、短期プロトタイプには過剰設計になりがちです。チームの習熟度とプロダクトの複雑さを秤にかけて判断しましょう。
